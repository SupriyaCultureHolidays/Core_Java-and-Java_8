JavaScript interpreted — সরাসরি browser বা Node-এ চলে। Java প্রথমে compile হয়, তারপর JVM interpret করে।

Execution pipeline

Bytecode (.class)   ← platform independent
     ↓  [JVM — Java Virtual Machine]

শব্দকী
JDKJava Development Kit — সম্পূর্ণ toolkit (compiler + JRE)। Java লিখতে এটি install করো।
JREJava Runtime Environment — JVM + libraries। Java app চালাতে end user এটি install করে।

মূল বিষয়: "Write Once, Run Anywhere" — .class bytecode যেকোনো OS-এ চলে, যেখানে JVM আছে।

03

Type Conversion

এক Data Type-এর value-কে অন্য Data Type-এ পরিবর্তন করাকে Type Conversion বলে। Java-তে এটি দুই ধরনের — স্বয়ংক্রিয় (Widening) এবং ম্যানুয়াল (Narrowing/Casting)।

প্রকার ১ — Implicit / Widening (স্বয়ংক্রিয়)

ছোট Data Type থেকে বড় Data Type-এ Java নিজেই Convert করে। কোনো Casting লিখতে হয় না।

byte → short → int → long → float → double
int → double (স্বয়ংক্রিয়)
int num = 25;
double d = num;          // cast লাগবে না
System.out.println(d);   // 25.0
char → int (Unicode value)
char ch = 'A';
int value = ch;
System.out.println(value);   // 65  ('A'-এর ASCII value = 65)

প্রকার ২ — Explicit / Narrowing (Casting)

বড় Data Type থেকে ছোট Data Type-এ Convert করতে Java নিজে করে না — আমাদের cast লিখতে হয়।

Syntax
smallerType variable = (smallerType) value;
double → int (দশমিক কেটে যায়)
double d = 10.8;
int num = (int) d;
System.out.println(num);   // 10  — দশমিক অংশ বাদ যায়, Round হয় না
int → char
int x = 66;
char ch = (char) x;
System.out.println(ch);   // B  (Unicode 66 = 'B')

Overflow — Range-এর বাইরে গেলে কী হয়

byte-এর Range হলো -128 থেকে 127। এর বাইরের মান রাখতে গেলে Overflow হয়।

Overflow উদাহরণ
int num = 130;
byte b = (byte) num;
System.out.println(b);    // -126  (byte overflow)

int num2 = 260;
byte b2 = (byte) num2;
System.out.println(b2);   // 4  (260 % 256 = 4)

Overflow — ফলাফল বের করার নিয়ম

দুটো ধাপ। প্রথমে value % 256 করো। তারপর যদি ফল ≥ 128 হয়, তাহলে 256 বিয়োগ করো।

Formula walkthrough
130 % 256 = 130  →  130 ≥ 128  →  130 - 256 = -126
140 % 256 = 140  →  140 ≥ 128  →  140 - 256 = -116
260 % 256 =   4  →    4 < 128  →  ফলাফল     =    4
256 % 256 =   0  →    0 < 128  →  ফলাফল     =    0

Arithmetic-এ Type Promotion

byte, short, বা char নিয়ে Arithmetic করলে Java সেগুলোকে আগে int-এ Promote করে।

byte + byte → int হয়ে যায়
byte a = 10;
byte b = 20;

int result = a + b;    // ✅ সঠিক
// byte result = a + b;  ❌ compile error — ফলাফল int হয়

System.out.println(result);   // 30
কেন? Java যেকোনো Arithmetic-এর আগে byte/short/char-কে int-এ বানিয়ে নেয়। byte ফেরত চাইলে cast করতে হবে: byte c = (byte)(a + b);

Integer Division-এর ফাঁদ

সাধারণ ভুল
int a = 10;
double b = a / 3;          // 3.0  — আগে Integer Division হয়!

double c = a / 3.0;        // 3.3333...  — 3.0 লিখলে double division হয়
System.out.println(c);

Boolean — Convert করা যায় না

Invalid — compile error
boolean flag = true;
int x = flag;    // ❌ boolean কোনো numeric type-এ convert হয় না

char ++ কৌশল

char increment করা
char c = 'A';
c++;
System.out.println(c);   // B  (65 → 66 → 'B')

Autoboxing এবং Unboxing (Java 5+)

Primitive ↔ Wrapper — স্বয়ংক্রিয় রূপান্তর
int num = 10;
Integer obj = num;   // Autoboxing — primitive থেকে Wrapper, স্বয়ংক্রিয়
int num2 = obj;      // Unboxing — Wrapper থেকে primitive, স্বয়ংক্রিয়

// Collections শুধু Object নেয়, তাই Wrapper লাগেই
List<Integer> list = new ArrayList<>();
list.add(10);              // ভেতরে ভেতরে autoboxing হয়ে যায়
int first = list.get(0);   // unboxing

String ↔ Number রূপান্তর

parseXxx() এবং valueOf()
String s = "123";
int n = Integer.parseInt(s);       // String → int
double d = Double.parseDouble("12.5");

int x = 45;
String str1 = String.valueOf(x);   // int → String
String str2 = Integer.toString(x); // int → String (বিকল্প)
মনে রাখো: Integer.parseInt("abc")-এর মতো non-numeric String দিলে NumberFormatException হয় — আগের Exception chapter-এ এটা দেখেছ।

Integer Caching — একটা চমকপ্রদ ফাঁদ

== দিয়ে Wrapper compare করার বিপদ
Integer a = 100;
Integer b = 100;
System.out.println(a == b);   // true! -128 থেকে 127 পর্যন্ত Java cache করে রাখে

Integer c = 200;
Integer d = 200;
System.out.println(c == d);        // false! cache range-এর বাইরে, দুটো আলাদা object
System.out.println(c.equals(d));   // true — value compare করতে .equals() ব্যবহার করো
নিয়ম: Wrapper Object compare করতে সবসময় == নয়, .equals() ব্যবহার করো — == reference compare করে, cache range-এর বাইরে গেলে ভুল ফলাফল দেবে।

Class এবং Object

এটি Java-র মূল ভিত্তি। Class হলো blueprint; Object হলো সেই blueprint থেকে তৈরি আসল জিনিস।

JavaScript
const calc = { add: (a,b) => a+b };
calc.add(3, 4);
Java
Calculator calc = new Calculator();
int result = calc.add(4, 5);
সম্পূর্ণ উদাহরণ
class Calculator {
    public int add(int n1, int n2) {
        return n1 + n2;
    }
}

class Demo {
    public static void main(String[] args) {
        Calculator calc = new Calculator();
        System.out.println(calc.add(4, 5));  // 9
    }
}
মূল পয়েন্ট: প্রতিটি Object-এর নিজস্ব instance variable আছে। obj.num বদলালে obj1.num বদলায় না।

Stack এবং Heap Memory

Java program run করার সময় JVM দুটো আলাদা Memory area ব্যবহার করে — Stack এবং Heap। এই দুটোর কাজ আলাদা এবং একসাথে কাজ করে।

সহজ Rule:
Stack = Local Variables + Method Calls + Reference Variables
Heap = Objects (new দিয়ে তৈরি সব কিছু)

Stack Frame — method call হলে কী হয়

প্রতিটি method call হলে Stack-এ একটি নতুন frame তৈরি হয়। Stack LIFO — শেষে যে method এসেছে সে সবার উপরে থাকে। Method শেষ হলে তার frame মুছে যায়।

Multiple Inheritance এবং Diamond Problem

Multiple Inheritance মানে একটি class একাধিক parent থেকে extend করে। Java-তে class দিয়ে multiple inheritance নেই — Diamond Problem-এর কারণে।

class A { void show() {} } class B { void show() {} } class C extends A, B { } ← COMPILE ERROR — Java allow করে না কারণ: C.show() call হলে Java কোনটা নেবে — A-র নাকি B-র? এটাই Diamond Problem।

d.swim();
মনে রাখো: Class → একটিমাত্র parent (extends)। Interface → যতগুলো খুশি (implements)।

Parent class-এর method child class-এ একই নাম, একই parameter দিয়ে নতুনভাবে লেখা। Child-এর version চলবে, parent-এরটা নয়।

Spring Boot Interview-তে প্রায়ই আসে — Java OOP-এর সবচেয়ে গুরুত্বপূর্ণ topic-গুলোর একটি।

Polymorphism

এটাই OOP-এর সবচেয়ে গুরুত্বপূর্ণ concept। যদি Inheritance + Method Overriding বুঝে থাকো, তাহলে Polymorphism খুব সহজ।

Polymorphism মানে কী?

Poly = অনেক (Many), Morph = রূপ (Forms)। অর্থাৎ — একই Method Call, কিন্তু Object অনুযায়ী আলাদা Behavior।

Real Life Example — Remote-এর Power Button একটাই → TV হলে : TV ON → AC হলে : AC ON → Projector হলে : Projector ON Button একটাই। কাজ Object অনুযায়ী আলাদা। এটাই Polymorphism।

Java Example

Parent ও দুইটা Child Class
class Animal {
    void sound() { System.out.println("Animal Sound"); }
}

class Dog extends Animal {
    @Override void sound() { System.out.println("Woof"); }
}

class Cat extends Animal {
    @Override void sound() { System.out.println("Meow"); }
}

আসল Polymorphism শুরু এখান থেকে

Animal a = new Dog(); — এটা একটু Strange লাগে। মনে হয়, Animal Variable-এ Dog Object কীভাবে সম্ভব? কারণ Dog IS-A Animal — Dog হচ্ছে একটা Animal, তাই এই assignment valid।

এবার a.sound(); call করলে Java কি Animal Sound print করবে? না। Output হয় Woof — কারণ Memory-তে আসল Object হচ্ছে Dog। Reference Animal হলেও Object Dog, আর Runtime-এ Java Object দেখে Method Call করে। এটাই Runtime Polymorphism।

Polymorphism না থাকলে?

ধরো ১০০টা Animal রাখতে হবে। Polymorphism ছাড়া প্রতিটা Type-এর জন্য আলাদা Variable লাগবে — অনেক Code লিখতে হবে। Polymorphism থাকলে Animal[] animals — একটাই Array-তে সব Animal রাখা যায়।

Reference vs Object — Memory-তে আসলে কী ঘটে

Animal a = new Dog(); — এটাকে দুই ভাগে ভাগ করে বুঝি। বাম পাশে Animal a মানে Stack-এ একটা Animal টাইপের Reference Variable। ডান পাশে new Dog() মানে Heap-এ আসল Dog Object তৈরি হচ্ছে। a টাইপে Animal হলেও, সে আসলে Dog Object-কে Point করছে।

STACK HEAP a ------------------> Dog Object খেয়াল করো: a Animal টাইপের হলেও সে Dog Object-কে Point করছে।

এটা Allowed কারণ Dog IS-A Animal (Inheritance সম্পর্ক)। যেমন — ল্যাব্রাডর IS-A Dog, Dog IS-A Animal। তাই Dog Object-কে Animal Reference-এ রাখা যায়, কিন্তু উল্টোটা (Animal IS-A Dog) সবসময় সত্যি নয়।

এবার a.sound(); কল করলে Java Reference-এর Type (Animal) নয়, আগে দেখে Heap-এ আসল Object কী — সেখানে Dog Object আছে, তাই Dog-এর Method চলে, Output হয় Woof।

Method Call-এর সময় Java Reference-এর Type দেখে না — Heap-এ থাকা Actual Object-এর Type দেখে।

Dynamic Method Dispatch

Parent type-এর reference দিয়ে Child object hold করা হলে, method call-এর সময় কোন class-এর method চলবে সেটা runtime-এ decide হয় — compile time-এ নয়।

Reference type vs Object type
Animal a = new Dog();    // reference type: Animal | object type: Dog

a.sound();   // কোনটা চলবে? — Animal-এর নাকি Dog-এর?
             // উত্তর: Dog-এর। Object type দেখে সিদ্ধান্ত হয়, reference type নয়।
STACK
main()
a:Animal ref →
HEAP
Dog object ← actual type এটা
a.sound() → Dog-এর sound() চলে
একটা method, object type অনুযায়ী আলাদা ফলাফল
Animal a;

a = new Dog();
a.sound();   // Woof! — Dog-এর method

a = new Cat();
a.sound();   // Meow! — Cat-এর method

// একই variable a, কিন্তু runtime-এ ভিন্ন method — এটাই Dynamic Dispatch
সম্পূর্ণ উদাহরণ
class Animal {
    void sound() {
        System.out.println("Animal");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("Woof");
    }
}

class Cat extends Animal {
    @Override
    void sound() {
        System.out.println("Meow");
    }
}

class Demo {
    public static void main(String[] args) {
        Animal a = new Dog();
        a.sound();   // Woof

        a = new Cat();
        a.sound();   // Meow
    }
}
শুধু overridden method-এর ক্ষেত্রে প্রযোজ্য। Child-এর নিজস্ব method (যেটা parent-এ নেই) Animal reference দিয়ে call করা যাবে না।

কেন a.bark() Call করা যায় না?

Reference Type-এর বাইরের method — Compile Error

    void sound() { }
}

class Dog extends Animal {
    void sound() { }
    void bark() { }   // শুধু Dog-এ আছে, Animal-এ নেই
}

// Main-এ:

Compiler Compile-time-এ শুধু Reference Type দেখে method আছে কিনা check করে। Runtime-এ JVM Object Type দেখে কোন version চলবে ঠিক করে — কিন্তু method-টা Reference Type-এর class-এ declare করা থাকতেই হবে।

Polymorphism আর Dynamic Method Dispatch-এর সম্পর্ক:
Polymorphism হলো Concept, Dynamic Method Dispatch হলো সেই Concept JVM-এ কাজ করার Mechanism।

Method Overriding → Runtime Polymorphism → Dynamic Method Dispatch (JVM runtime-এ ঠিক করে কোন overridden method চলবে)

final keyword

final মানে "আর পরিবর্তন করা যাবে না" — এই একটা কথা মনে রাখলেই যথেষ্ট। final তিন জায়গায় ব্যবহার করা যায়: Variable, Method এবং Class।

1. final Variable

একবার Value Assign করার পর সেই Variable আর Change করা যায় না — একে Constant বলে।

final Variable — Constant
public class Demo {
    public static void main(String[] args) {
        final double PI = 3.14159;
        System.out.println(PI);

        // PI = 3.14;  ← ❌ Compile Error, একবার Value পেলে আর Change করা যাবে না
    }
}
Output: 3.14159

যেমন final int MAX_AGE = 150; বা final double PI = 3.14159; — এগুলো সব Constant, যাদের Value কখনো বদলাবে না।

2. final Method

Parent Class যদি কোনো Method-কে final করে দেয়, Child Class সেটা Override করতে পারবে না।

final Method — Override করা যাবে না
class Animal {
    final void breathe() {
        System.out.println("Breathing");
    }
}

class Dog extends Animal {
    void bark() {
        System.out.println("Woof");
    }   // ঠিক আছে — bark() নতুন method

    // @Override
    // void breathe() { }   ← ❌ Compile Error, final method override করা যায় না
}
কেন দরকার? ধরো Bank Software-এ final void calculateGST() { } — কোনো Developer যেন GST Calculation-এর Logic বদলে দিতে না পারে, তাই এই Method-কে final করা হয়।

3. final Class

final Class-কে কেউ Extend করতে পারবে না।

final Class — Extend করা যাবে না
final class Animal {
}

class Dog extends Animal {   // ❌ Compile Error — final class extend করা যায় না
}

String Class final কেন?

প্রতিদিন যে String name = "Java"; লেখো, সেই String Class-টি নিজেই final — অর্থাৎ class MyString extends String সম্ভব না।

String Immutable, Secure ও Reliable রাখার জন্য এটাকে final করা হয়েছে। String Extend করা গেলে কেউ equals(), hashCode(), length()-এর মতো Method বদলে দিতে পারত, আর তাতে Java-এর অনেক কিছু ভেঙে যেত।

toString() — Object-কে String-এ দেখাও

class Student {

equals() — value দিয়ে তুলনা করো

Default equals() এবং override করা
Student s1 = new Student("Rahul", 90);
Student s2 = new Student("Rahul", 90);

System.out.println(s1 == s2);        // false — আলাদা object
System.out.println(s1.equals(s2));   // false — default equals() reference compare করে

// equals() override করো
@Override
public boolean equals(Object obj) {
    if (this == obj) return true;
    if (!(obj instanceof Student)) return false;
    Student other = (Student) obj;
    return this.name.equals(other.name) && this.marks == other.marks;
}

System.out.println(s1.equals(s2));   // true — এখন value compare করছে

equals() ও hashCode() Contract

আগের Chapter-এ equals() ও hashCode() override করা শিখেছি। কিন্তু শুধু override করলেই হয় না — একটা নির্দিষ্ট Contract (নিয়ম) মেনে চলতে হয়, নাহলে HashMap/HashSet ভুল আচরণ করবে।

equals()-এর ৫টি নিয়ম

নিয়মমানে
Reflexivex.equals(x) সবসময় true
Symmetricx.equals(y) true হলে y.equals(x)-ও true
Transitivex.equals(y) ও y.equals(z) true হলে x.equals(z)-ও true
ConsistentObject পরিবর্তন না হলে বারবার call করলে একই ফলাফল
Null-এর সাথেx.equals(null) সবসময় false

hashCode()-এর মূল নিয়ম

যদি দুটো Object equals()-এ true হয়, তাহলে তাদের hashCode() অবশ্যই সমান হতে হবে। উল্টোটা সত্যি না — দুটো Object-এর hashCode() সমান হলেও equals() false হতে পারে (একে Collision বলে)।

Contract মেনে চলা
Student s1 = new Student("Rahul", 90);
Student s2 = new Student("Rahul", 90);

System.out.println(s1.equals(s2));      // true
System.out.println(s1.hashCode() == s2.hashCode());  // অবশ্যই true হতে হবে

Contract ভাঙলে কী হয় — HashSet-এ ভুতুড়ে Duplicate

শুধু equals() override, hashCode() ভুলে গেলে
class Student {
    String name;
    Student(String name) { this.name = name; }

    @Override
    public boolean equals(Object o) {
        if (!(o instanceof Student)) return false;
        return name.equals(((Student) o).name);
    }
    // hashCode() override করা হয়নি — default (memory address ভিত্তিক) থেকে যাবে
}

Set<Student> set = new HashSet<>();
set.add(new Student("Rahul"));
set.add(new Student("Rahul"));

System.out.println(set.size());   // 2! — অথচ equals() true, তবু duplicate ধরা পড়লো না

কারণ HashSet আগে hashCode() দিয়ে bucket খোঁজে — দুটো Object-এর hashCode() আলাদা হওয়ায় তারা আলাদা bucket-এ চলে যায়, equals() পর্যন্ত তুলনাই হয় না।

Object Cloning

একটি existing Object-এর হুবহু কপি তৈরি করাকে Cloning বলে — new keyword দিয়ে নতুন Object না বানিয়ে, আরেকটা Object-এর state কপি করে নতুন Object বানানো।

Cloneable Interface ও clone() Method

Java
class Student implements Cloneable {
    String name;
    int marks;
    Student(String name, int marks) { this.name = name; this.marks = marks; }

    @Override
    public Object clone() throws CloneNotSupportedException {
        return super.clone();
    }
}

Student s1 = new Student("Rahul", 90);
Student s2 = (Student) s1.clone();

System.out.println(s1 == s2);              // false — আলাদা object
System.out.println(s1.name.equals(s2.name)); // true — same value কপি হয়েছে
Cloneable একটা Marker Interface (আগের Chapter মনে আছে?) — কোনো method নেই, শুধু tag। এটা implement না করে clone() call করলে CloneNotSupportedException হয়।

Shallow Copy — ডিফল্ট আচরণ

Object-এর clone() শুধু Shallow Copy করে — Primitive field কপি হয়, কিন্তু Object reference field (array, অন্য class) একই reference শেয়ার করে।

Shallow Copy-এর বিপদ
class Address {
    String city;
    Address(String city) { this.city = city; }
}

class Student implements Cloneable {
    String name;
    Address address;
    Student(String name, Address address) { this.name = name; this.address = address; }

    @Override
    public Object clone() throws CloneNotSupportedException {
        return super.clone();   // শুধু shallow copy
    }
}

Student s1 = new Student("Rahul", new Address("Dhaka"));
Student s2 = (Student) s1.clone();

s2.address.city = "Chittagong";
System.out.println(s1.address.city);   // Chittagong! — দুজনেই একই Address object শেয়ার করছে

Deep Copy — সমাধান

clone()-এর ভেতরে Address-ও আলাদা কপি করা
@Override
public Object clone() throws CloneNotSupportedException {
    Student copy = (Student) super.clone();
    copy.address = new Address(this.address.city);   // নতুন Address object
    return copy;
}

Student s2 = (Student) s1.clone();
s2.address.city = "Chittagong";
System.out.println(s1.address.city);   // Dhaka — এখন আর প্রভাব পড়ে না

Copy Constructor — আধুনিক বিকল্প

বাস্তব প্রজেক্টে Cloneable/clone()-এর CloneNotSupportedException ও casting-এর ঝামেলা এড়াতে অনেকে সরাসরি Copy Constructor ব্যবহার করে।

Java
class Student {
    String name;
    Address address;

    Student(Student other) {                        // Copy Constructor
        this.name = other.name;
        this.address = new Address(other.address.city);
    }
}

Student s2 = new Student(s1);   // clone() লাগলো না

Immutable Class

Immutable Class মানে এমন class, যার Object তৈরি হওয়ার পর তার state (field values) আর কখনো বদলানো যায় না। String এই নিয়মেরই সবচেয়ে বড় উদাহরণ — আগে দেখেছ String কেন Immutable রাখা হয়েছে।

কেন দরকার

Thread-safety — একাধিক Thread একসাথে একই Object read করলেও কোনো সমস্যা নেই, কেউ modify করতে পারে না। HashMap key হিসেবে নিরাপদ — key-এর hashCode() কখনো বদলায় না। Caching ও Sharing-ও নিরাপদ হয়।

Immutable Class বানানোর নিয়ম

Singleton Class

Singleton Design Pattern নিশ্চিত করে একটা class-এর সম্পূর্ণ Application-এ মাত্র ১টা Object তৈরি হবে — যেমন Database Connection, Logger, Configuration।

  2. Private static instance variable — নিজের একমাত্র Object ধরে রাখে

class Logger {

Upcasting এবং Downcasting

এই Chapter-টা আসলে Dynamic Method Dispatch-এর পরের ধাপ। যদি Animal a = new Dog(); বুঝে থাকো, তাহলে এটাও সহজ।

Casting মানে কী?

Casting মানে Reference Type পরিবর্তন করা। Object পরিবর্তন হয় না — শুধু Reference-এর Type পরিবর্তন হয়।

Hierarchy
        Object
           ↑
        Animal
           ↑
          Dog

Dog সবচেয়ে নিচে, Animal উপরে।

আগে পুরো Example-টা দেখে নাও — নিচের সব ব্যাখ্যা এই একই Code নিয়ে

নিচের সব Code fragment, Memory diagram, এবং ব্যাখ্যা — সব এই একই Animal / Dog Class দুটো ধরে করা হয়েছে। এই পুরো Code-টা মাথায় রেখে নিচের অংশগুলো পড়ো।

Java — এই Chapter-এর মূল Example
class Animal {
    void sound() {
        System.out.println("Animal Sound");
    }
}

class Dog extends Animal {
    @Override
    void sound() {
        System.out.println("Woof");
    }

    void bark() {
        System.out.println("Bark Bark");
    }
}

public class Demo {
    public static void main(String[] args) {
        Dog d = new Dog();
        Animal a = d;   // Upcasting

        a.sound();      // Output: Woof
        // a.bark();  ← COMPILE ERROR

        Dog d2 = (Dog) a;  // Downcasting
        d2.bark();         // Output: Bark Bark
    }
}

নিচে এই Code-টাকেই টুকরো টুকরো করে (line by line) ব্যাখ্যা করা হলো।

1. Upcasting — Child → Parent

Upcasting — automatic
Dog d = new Dog();
Animal a = d;   // এর সমান: Animal a = (Animal) d; কিন্তু (Animal) লিখতে হয় না

Upcasting মানে Dog → Animal। এটা Java নিজেই করে, তাই একে Automatic Upcasting বলে।

STACK
main()
d = ref →
a = ref → // একই ঠিকানা
HEAP
Dog object  ← d ও a দুজনেই এখানে

খেয়াল করো, Object এখনও Dog। শুধু Reference Type পরিবর্তন হয়েছে।

a.sound() — চলবে
a.sound();   // Output: Woof — Heap-এ Dog Object আছে, Dynamic Dispatch
a.bark() — Compile Error
a.bark();   // COMPILE ERROR — Compiler শুধু Reference Type (Animal) দেখে, Animal-এ bark() নেই

sound() চলে, bark() চলে না কেন — আলাদা নিয়ম না, একই নিয়ম দুই ধাপে

আলাদা নিয়ম না — আসলে একই নিয়ম, দুই ধাপে কাজ করে। শুধু sound() আর bark()-এর ক্ষেত্রে ধাপ দুটোর ফলাফল আলাদা হয়ে যায়।

ধাপ ১ — Compile Time (Compiler-এর কাজ)
Compiler প্রথমে চেক করে: reference type-এর Class-এ (এখানে Animal) এই method-টা আছে কিনা? এটা শুধু Class-এর blueprint দেখে, Object দেখে না — কারণ compile time-এ কোনো Object তৈরিই হয়নি এখনো।
  • sound() → Animal Class-এ আছে। ✅ Compile পাস।
  • bark() → Animal Class-এ নেই। ❌ Compile Error — এখানেই থেমে যায়, Runtime পর্যন্ত পৌঁছায় না।
ধাপ ২ — Runtime (JVM-এর কাজ)
যদি ধাপ ১ পাস হয়, তাহলে JVM দেখে: Heap-এ Actual Object আসলে কী (Dog)? সেই Object-এর version-টাই চালায় (Dynamic Dispatch)।
  • sound() ধাপ ১ পাস করেছে, তাই ধাপ ২-তে যায় → Heap-এ Dog Object, তাই Dog-এর overridden sound() চলে (Output: Woof)।
  • bark() ধাপ ১-তেই আটকে গেছে, তাই ধাপ ২ পর্যন্ত পৌঁছায়ইনি।

সারমর্ম: Compiler শুধু "reference type-এ method declare করা আছে কিনা" — এটা দেখে অনুমতি দেয়। অনুমতি পেলে, কোন Class-এর version চলবে সেটা ঠিক করে Actual Object (Runtime)। bark()-এর ক্ষেত্রে অনুমতিই মেলেনি, তাই runtime dispatch-এর প্রশ্নই আসে না।

2. Downcasting — Parent → Child

Downcasting — manual cast
Animal a = new Dog();

Dog d = (Dog) a;   // Cast লিখতে হবে
d.bark();          // এখন চলবে

Downcasting মানে Animal → Dog। এখানে Cast লিখতে হয়, কারণ Java নিজে Risk নেয় না।

STACK
main()
a = ref →
d = ref → // একই ঠিকানা
HEAP
Dog object  ← a ও d দুজনেই এখানে

দুইটা Reference একই Dog Object-কে দেখছে।

কেন Manual? — ভুল Downcasting-এর ঝুঁকি

কারণ সব Animal Dog নয়। ধরো Animal a = new Cat(); — এখন Dog d = (Dog) a; লিখলে Compiler বলবে "ঠিক আছে, তুমি Cast করতে চাইছ।" কিন্তু Runtime-এ দেখে Actual Object = Cat, Dog নয়। তখন ClassCastException হয়।

ভুল Downcasting — ClassCastException
Animal a = new Cat();
Dog d = (Dog) a;   // RUNTIME ERROR: ClassCastException

instanceof দিয়ে সতর্কতা

আগে Check করো Object আসলে Dog কিনা, তারপর Cast করো।

instanceof দিয়ে safe downcast
if (a instanceof Dog) {

}

Animal a = new Dog();

Animal a2 = new Cat();

Abstract keyword

abstract class সরাসরি instantiate করা যায় না — object তৈরি হয় না। কিন্তু child class-এর মাধ্যমে কাজ করানো যায়। Abstract method-এর কোনো body নেই — child-কে implement করতেই হবে।

Chain of Abstract Classes — সব Abstract Method একসাথে Implement করা বাধ্যতামূলক না

একটা abstract class অন্য একটা abstract class-কে extend করলে, তাকে বাবার সব abstract method implement করতে হবে — এমন কোনো নিয়ম নেই। যেটা implement করেনি, সেটা নিজের মধ্যেও abstract-ই থেকে যাবে। শুধু concrete class-কেই (যেটার object বানানো হবে) বাকি সব abstract method implement করতে হয়।

Car → WagnoR → UpdateWagnoR
abstract class Car {
    public abstract void drive();
    public abstract void fly();

    public void playMusic() {                    // concrete method
        System.out.println("play music");
    }

    public void printReceipt(double amount) {     // concrete method — parameter নিয়ে কাজ করে
        System.out.println("Receipt: Paid " + amount);
    }
}

abstract class WagnoR extends Car {     // এটাও abstract থাকতে বাধ্য
    public void drive() {               // শুধু drive() implement করলো
        System.out.println("Driving...");
    }
    // fly() implement করেনি — তাই WagnoR-কেও abstract রাখতে হলো
}

class UpdateWagnoR extends WagnoR {     // concrete class
    public void fly() {                 // বাকি থাকা fly() implement করলো
        System.out.println("flying...");
    }
}

public class Demo {
    public static void main(String[] args) {

        // Car obj = new Car();       ← COMPILE ERROR — Car abstract
        // Car obj = new WagnoR();    ← COMPILE ERROR — WagnoR-ও abstract (fly() নেই)
        Car obj = new UpdateWagnoR();  // Upcasting — এটাই একমাত্র concrete class

        obj.drive();               // Output: Driving...          (WagnoR-এর version)
        obj.playMusic();           // Output: play music          (Car-এর concrete method)
        obj.printReceipt(500);     // Output: Receipt: Paid 500.0 (Car-এর concrete method, parameter সহ)
    }
}

obj.drive() চলে WagnoR-এর version (Dynamic Dispatch — Heap-এ Actual Object UpdateWagnoR, drive() সেখানে override করা হয়নি তাই chain বেয়ে উপরে WagnoR-এর version খুঁজে পায়)। obj.playMusic() চলে Car-এর concrete method, কারণ সেটা কোথাও override হয়নি।

@Override — Compile-time Safety

@Override লিখে compiler-কে বলা হয়, "আমি Parent-এর Method Override করছি" — compiler তখন যাচাই করে দেখে যে Parent class-এ সত্যিই এমন নামের method আছে কিনা।

class Animal {

class Dog extends Animal {

Functional Interface

Java 8-এর গুরুত্বপূর্ণ একটা Topic। Definition: যে Interface-এ Exactly (ঠিক) ১টা Abstract Method থাকে, তাকে Functional Interface বলে।

Normal Interface vs Functional Interface

❌ Normal Interface (2টা Abstract Method)
interface Animal {
    void eat();
    void sleep();
}
// মোট Abstract Method = ২টা → Functional Interface না
✅ Functional Interface (1টা Abstract Method)
@FunctionalInterface
interface Greeting {
    void greet(String name);   // শুধু একটাই abstract method

Thread States

একজন মানুষের জীবনের মতো, একটা Thread-এরও জন্ম, কাজ করা, অপেক্ষা করা এবং শেষ হওয়ার বিভিন্ন state আছে। এটাই Thread Life Cycle।

পুরো Flow
            NEW
             │
         start()
             │
             ▼
         RUNNABLE
             │
        CPU পেলে
             ▼
          RUNNING
      ┌──────┼─────────┐
      │      │         │
  sleep()  join()  synchronized lock না পেলে
      │      │         │
      ▼      ▼         ▼
TIMED_WAITING WAITING BLOCKED
      │      │         │

             │

             ▼

             ▼

১. NEW

Thread object তৈরি হয়েছে, কিন্তু start() হয়নি।

NEW state
MyThread t = new MyThread();   // State: NEW — start() হয়নি

২. RUNNABLE (Ready, Running নয়)

t.start() কল করলে thread RUNNABLE-এ যায় — মানে "CPU পেলেই কাজ শুরু করব"। অনেকে ভুল করে RUNNABLE-কে Running ভাবে — আসলে এটা শুধু Ready। তিনজন student ইন্টারভিউ রুমের বাইরে অপেক্ষা করছে (RUNNABLE), কিন্তু ভিতরে একজনই ঢুকতে পারবে (RUNNING)।

RUNNABLE state
t.start();   // State: RUNNABLE — Ready, CPU-র জন্য অপেক্ষা

৩. RUNNING

CPU যখন thread-কে actually execute করছে, তখন সে run()-এর ভিতরের কাজ করে।

run() চলছে
public void run(){
    for(int i=1;i<=5;i++){
        System.out.println(i);   // State: RUNNING
    }
}

৪. WAITING

এক thread অন্য thread-এর জন্য অপেক্ষা করলে। সবচেয়ে common কারণ: join()।

join() → WAITING
t1.start();
t1.join();     // Main thread এখানে WAITING — t1 শেষ না হওয়া পর্যন্ত
t2.start();

৫. TIMED_WAITING

নির্দিষ্ট সময়ের জন্য pause। সবচেয়ে common কারণ: Thread.sleep(ms)।

sleep() → TIMED_WAITING
for(int i=1;i<=5;i++){
    System.out.println(i);
    Thread.sleep(1000);   // State: TIMED_WAITING — ১ সেকেন্ড ঘুমাও
}

৬. BLOCKED

synchronized-এর জন্য হয়। Thread-1 lock নিয়ে method-এর ভিতরে কাজ করছে; Thread-2 ওই lock না পাওয়া পর্যন্ত BLOCKED হয়ে থাকে।

synchronized lock → BLOCKED
synchronized void increment(){
    // Thread-1 এখানে Lock পেয়ে Working
    // Thread-2 Lock না পেয়ে BLOCKED
}

৭. TERMINATED

run() শেষ হয়ে গেলে thread TERMINATED — আর restart করা যায় না।

run() শেষ → TERMINATED
public void run(){
    System.out.println("Hello");   // Print হয়ে গেল, run() শেষ
}
// State: TERMINATED

Interview-এর গুরুত্বপূর্ণ নোট: বাস্তবে JVM-এর Thread.State enum-এ আলাদা RUNNING state নেই — OS scheduler CPU দিচ্ছে কি না, সেটাও RUNNABLE-এর মধ্যেই ধরা হয়। শেখার সুবিধার জন্য বেশিরভাগ বই/শিক্ষক NEW → RUNNABLE → RUNNING → WAITING/TIMED_WAITING/BLOCKED → TERMINATED flow দেখান, যা conceptually ঠিক, কিন্তু কেউ যদি জিজ্ঞেস করে "Java API-তে RUNNING state আছে কি?" — উত্তর: না, CPU-তে execute হওয়া thread-ও RUNNABLE-এর অংশ।

আগে Problem-টা বুঝি

এই সমস্যার সমাধানেই দুটো interface — Comparable (class নিজে বলে দেয় কীভাবে sort হবে) আর Comparator (বাইরে থেকে বলা হয় কীভাবে sort করতে হবে)।

Comparable — class নিজেই natural order define করে

compareTo() — একটাই "natural" sorting rule
class Student implements Comparable<Student> {
    String name; int marks;
    Student(String name, int marks) { this.name=name; this.marks=marks; }

    @Override
    public int compareTo(Student other) {
        return this.marks - other.marks;  // marks অনুযায়ী ascending
    }
}

Collections.sort(list);  // Java নিজেই compareTo() call করবে
// Output: Rahim 70, Sumi 80, Karim 90

Comparator — বাইরে থেকে custom sort

আজ marks দিয়ে sort করলে, কাল name দিয়ে, পরশু descending — এসবের জন্য প্রতিবার Student class change করতে হবে না। Class-এর বাইরে থেকে rule দেওয়া হয়, এটাই Comparator।

compare() — যত খুশি sorting rule
// Name দিয়ে ascending
list.sort((s1, s2) -> s1.name.compareTo(s2.name));
// Output: Karim, Rahim, Sumi

// Marks দিয়ে descending
list.sort((s1, s2) -> s2.marks - s1.marks);
// Output: 90, 80, 70

// built-in helper method দিয়ে (readable, cleaner)
list.sort(Comparator.comparingInt(s -> s.marks));         // ascending
list.sort(Comparator.comparingInt((Student s) -> s.marks).reversed());  // descending

Serialization

কোনো Object-কে Byte-এর ধারায় রূপান্তর করে File বা Network-এ পাঠানোকে Serialization বলে, আর সেই Byte থেকে আবার Object ফিরিয়ে আনাকে Deserialization বলে। আগে Marker Interface chapter-এ Serializable-এর নাম শুনেছিলে — এখানে পুরোটা দেখব।

Object লেখা — ObjectOutputStream

Java
class Student implements Serializable {
    String name;
    int marks;
    Student(String name, int marks) { this.name = name; this.marks = marks; }
}

Student s = new Student("Rahul", 90);

try (ObjectOutputStream out = new ObjectOutputStream(new FileOutputStream("student.ser"))) {
    out.writeObject(s);   // Object-টা file-এ byte আকারে লেখা হলো
}

Object পড়া — ObjectInputStream

Java
try (ObjectInputStream in = new ObjectInputStream(new FileInputStream("student.ser"))) {
    Student s2 = (Student) in.readObject();   // byte থেকে আবার Object
    System.out.println(s2.name + " " + s2.marks);   // Rahul 90
}

transient keyword — কোনো Field Serialize না করা

Java
class User implements Serializable {
    String username;
    transient String password;   // এই field Serialize হবে না, নিরাপত্তার জন্য
}

// Deserialize করার পর password = null থাকবে

serialVersionUID — Version মেলানো

Java
class Student implements Serializable {
    private static final long serialVersionUID = 1L;
    String name;
    int marks;
}
মনে রাখো: serialVersionUID না দিলে Java নিজে একটা generate করে, কিন্তু class-টা পরে পরিবর্তন হলে (field যোগ/বাদ) পুরনো Serialized file পড়তে গেলে InvalidClassException হতে পারে। তাই নিজে থেকে একটা fixed value দেওয়াই ভালো Practice।
